# GustoBot 七条业务线 · 一页纸产品说明

> **读者**：AI 产品经理 / 业务 / 测试  
> **用途**：写 PRD、培训客服测试、对齐「用户问什么 → 系统走哪条线」  
> **版本**：v1.0

---

## 总览：七条线一张表

| 业务线 | 内部代号 | 一句话 | 查库吗 | 主要数据 |
|--------|----------|--------|--------|----------|
| ① 闲聊 | general-query | 打招呼、道谢，直接聊 | 否 | 无 |
| ② 追问 | additional-query | 信息不够，先问清楚 | 否 | 无（仅范围判断） |
| ③ 文化知识 | kb-query | 历史、典故、传入史 | 是 | Milvus + Postgres |
| ④ 做菜图谱 | graphrag-query | 怎么做、要什么料、技巧 | 是 | Neo4j（+ 可选 LightRAG） |
| ⑤ 数据统计 | text2sql-query | 多少、排名、统计 | 是 | MySQL |
| ⑥ 图片 | image-query | 识图 / 生图 | 视情况 | 视觉 API / 生图 API |
| ⑦ 文件 | file-query | 上传文档再问答 | 上传后查 Milvus | 用户文件 |

**入口规则**：用户每发一条消息，系统先做 **分诊（路由）**，只进其中 **一条** 主业务线（上传图片/文件时优先走 ⑥/⑦）。

---

# ① 闲聊业务线

## 是什么

用户只是在寒暄、感谢、闲聊，**不需要查菜谱库**。系统用固定「厨友/亲～」风格直接回复。

## 适用问题（要进这条线）

- 「你好」「早上好」「谢谢」「再见」
- 短情绪句：「今天心情不错」
- **注意**：若明显在问怎么做、历史典故，不应进这条线

## 不适用

- 任何需要查菜谱、统计、文化知识的具体问题

## 工作流（3 步）

```
用户消息
  → ① 分诊：判定为 general-query
  → ② 闲聊 Prompt：生成亲切回复（带 logic 分类理由）
  → ③ 返回用户
```

## 装备清单

| 项目 | 有没有 |
|------|--------|
| 人设 Prompt | ✅ 有（下厨房风格） |
| 多步工作流 | ❌ 仅 1 步生成 |
| 数据库 | ❌ |
| 工具调用 | ❌ |

## 成功标准（产品）

- 回复礼貌、简短，有「厨友/亲」类称呼
- **不触发**任何检索、不出现「根据知识库…」
- 非菜谱话题可礼貌带过，不硬答

## 5 条测试题

| # | 输入 | 期望 |
|---|------|------|
| 1 | 你好 | 问候，不查库 |
| 2 | 谢谢你 | 感谢回应 |
| 3 | 早上好 | 问候 |
| 4 | 再见 | 告别 |
| 5 | 今天心情不错 | 闲聊，不强行推荐菜 |

# ② 追问业务线

## 是什么

用户 **想咨询菜谱，但没说清关键信息**（缺菜名、缺场景）。系统先 **友好追问**，而不是瞎猜或乱查库。

## 适用问题

- 「我想做菜」「怎么做好吃的」
- 「这个菜怎么做」（没说是哪道菜）
- 「这个菜热量高吗」（没说是哪道）
- 烹饪失败描述太模糊

## 不适用（应拒答而非追问）

- 明显无关：「今天天气怎么样」→ 护栏拒答，引导回菜谱

## 工作流（4 步）

```
用户消息
  → ① 分诊：additional-query
  → ② 范围护栏：是否与菜谱相关？
        · 无关 → 固定拒答文案
        · 相关 → 继续
  → ③ 追问 Prompt：一次只问一个关键问题（尽量简短）
  → ④ 返回用户
```

## 装备清单

| 项目 | 有没有 |
|------|--------|
| 人设 Prompt | ✅ 厨友语气 |
| 多步工作流 | ✅ 护栏 + 追问 |
| 数据库 | ❌ 不检索 |
| 工具调用 | ❌（护栏可能参考 Neo4j 结构说明，但不查用户题） |

## 成功标准

- 缺信息时 **明确追问**（菜名、场景等），一次一问
- 无关问题 **礼貌拒答**，不开始做菜教程
- 回复不宜过长（Prompt 要求约 20 字内倾向）

## 5 条测试题

| # | 输入 | 期望 |
|---|------|------|
| 1 | 我想做菜 | 追问想做什么菜/口味 |
| 2 | 这个菜怎么做 | 追问是哪道菜 |
| 3 | 这个菜热量高吗 | 追问菜名 |
| 4 | 今天天气怎么样 | **拒答**，非菜谱 |
| 5 | 我想做一道好吃的 | 追问具体偏好/菜名 |

# ③ 文化知识业务线（知识库 RAG）

## 是什么

回答 **饮食文化史**：典故、命名由来、食材何时传入、菜系发展等。从 **长文本知识库** 检索后再写答案，**不讲具体烹饪步骤**。

## 适用问题

- 「宫保鸡丁的历史典故」「佛跳墙的由来」
- 「蔬菜是什么时候传入中国的」「为什么叫胡萝卜」
- 「川菜的特点和历史背景」「铁锅何时普及」

## 不适用（应走其他线）

- 「红烧肉怎么做」→ ④ 做菜图谱
- 「有多少道菜」→ ⑤ 数据统计

## 工作流（5 步）

```
用户消息
  → ① 分诊：kb-query
  → ② KB 护栏：是否文化/历史范围？否 → 拒答
  → ③ KB 调度：选检索源（默认 Postgres 优先，Milvus 兜底）
  → ④ 检索：捞出相关段落
  → ⑤ KB 生成 Prompt：仅依据段落写文化解读 → 返回
```

## 装备清单

| 项目 | 有没有 |
|------|--------|
| 人设 Prompt | ✅ 文化讲解助手 |
| 多步工作流 | ✅ 护栏→调度→检索→生成 |
| 数据库 | ✅ Milvus（data/kb/data.txt）+ Postgres（结构化，需 ingest） |
| 工具调用 | ✅ 向量检索、可选 Rerank、可选外部搜索 |

## 数据从哪来

- 主文本：`data/kb/data.txt`（饮食文化长文，切段后入 Milvus）
- 结构化：Excel 等经 ingest 入 Postgres（若未配置则主要靠 Milvus）

## 成功标准

- 路由 **必须** 进 kb-query（不与「怎么做」混淆）
- 答案 **有文化/历史内容**，**无** 具体步骤、用量、医疗建议
- 检索为空时：诚实说「知识库暂未找到」，不编造

## 5 条测试题

| # | 输入 | 期望 |
|---|------|------|
| 1 | 宫保鸡丁的历史典故 | kb，讲历史，不讲步骤 |
| 2 | 蔬菜是什么时候传入中国的 | kb，传入史 |
| 3 | 为什么叫胡萝卜 | kb，命名/由来 |
| 4 | 佛跳墙的由来 | kb，典故 |
| 5 | 火星上的菜谱历史 | kb，应无结果不编造 |

# ④ 做菜图谱业务线（知识图谱 RAG）

## 是什么

回答 **可操作菜谱问题**：怎么做、要什么食材、步骤顺序、烹饪方法等。从 **Neo4j 知识图谱**（菜-食材-步骤-口味关系）查事实，再组织成用户能照着做的说明。

## 适用问题

- 「香肠炒菜干怎么做」「宫保鸡丁需要哪些食材」
- 「糖醋排骨用什么烹饪方法」「炒青菜怎么保持翠绿」
- 「怎么判断鱼熟了」（偏技巧，可能走 LightRAG）

## 不适用

- 「宫保鸡丁的典故」→ ③ 文化知识
- 「有多少道川菜」→ ⑤ 数据统计

## 工作流（6 步）

```
用户消息
  → ① 分诊：graphrag-query
  → ② 图谱护栏：是否菜谱/图谱可答？否 → 拒答
  → ③ 规划（Planner）：复杂题拆子任务（简单题可跳过）
  → ④ 选工具：预置查询 / 动态图谱查询 / LightRAG
  → ⑤ 执行：查 Neo4j（或 LightRAG）
  → ⑥ 汇总 Prompt：写成步骤/食材说明 → 返回
```

## 装备清单

| 项目 | 有没有 |
|------|--------|
| 人设 Prompt | ✅ 汇总阶段（亲切菜谱助手） |
| 多步工作流 | ✅ 最长之一 |
| 数据库 | ✅ Neo4j（由 recipe.json 导入的图） |
| 工具调用 | ✅ 预置 Cypher、动态 Cypher、LightRAG |

## 数据从哪来

- 源文件：`data/recipe.json`（原材料）
- 真正查询：**Neo4j** 里的节点与关系（Dish、Ingredient、步骤、口味等）

## 成功标准

- 路由 **必须** graphrag-query
- 答案含 **步骤或食材**（库里有则必须体现）
- 不编造不存在的菜或步骤；查不到应说明

## 5 条测试题

| # | 输入 | 期望 |
|---|------|------|
| 1 | 香肠炒菜干怎么做 | 有做法/步骤 |
| 2 | 宫保鸡丁需要哪些食材 | 列食材 |
| 3 | 红烧肉怎么做 | 有步骤（与「典故」区分） |
| 4 | 麻婆豆腐的主辅料有哪些 | 列主辅料 |
| 5 | 不存在的菜 XYZABC 怎么做 | 无结果不编造 |

# ⑤ 数据统计业务线（Text2SQL）

## 是什么

回答 **数字、数量、排名、占比** 类问题。把自然语言变成 **SQL**，查 **MySQL 菜谱统计表**，再用口语说出结果。

## 适用问题

- 「数据库里有多少道菜」「统计有多少道川菜」
- 「哪个菜系的菜谱最多」「麻辣口味的菜有多少」
- 「统计每个口味的菜品数量」

## 不适用

- 「川菜有哪些代表菜」（列举菜名）→ 通常 ④ 图谱，不是纯统计
- 「宫保鸡丁怎么做」→ ④

## 工作流（6 步）

```
用户消息
  → ① 分诊：text2sql-query
  → ② 进入图谱子工作流（与④共用入口，但选不同工具）
  → ③ 图谱护栏 → ④ 选工具：text2sql_query
  → ⑤ SQL 生成 + 执行 MySQL
  → ⑥ 汇总 Prompt：用自然语言报数字 → 返回
```

## 装备清单

| 项目 | 有没有 |
|------|--------|
| 人设 Prompt | ✅ 汇总阶段 |
| 多步工作流 | ✅ 与④共用子流程 |
| 数据库 | ✅ MySQL（recipe_db 等） |
| 工具调用 | ✅ Text2SQL（分析意图 + 生成 SQL） |

## 成功标准

- 含「多少/统计/排名」类问题 **优先** 进 text2sql-query
- 答案 **必须出现数字或明确统计结论**
- 不与「列举菜名」类问题混淆

## 5 条测试题

| # | 输入 | 期望 |
|---|------|------|
| 1 | 数据库里有多少道菜 | 有总数 |
| 2 | 统计有多少道川菜 | 有数字 |
| 3 | 哪个菜系的菜谱最多 | 有排名/菜系名+数量 |
| 4 | 麻辣口味的菜有多少 | 有数字 |
| 5 | 统计每个口味的菜品数量 | 有分组统计 |

# ⑥ 图片业务线

## 是什么

两种模式：**识图**（用户上传菜品图，问这是什么/热量等）和 **生图**（用户文字描述，生成菜品图）。

## 适用问题

- 识图：带图片 + 「这是什么菜」「这图里有哪些食材」
- 生图：「生成一张宫保鸡丁的图片」「画一份红烧肉」

## 工作流

**识图：**
```
上传图片 + 文字
  → ① 分诊：image-query（或有 image_path 强制进入）
  → ② 视觉 API：生成图片描述
  → ③ 图片回复 Prompt：结合描述答用户 → 返回
```

**生图：**
```
纯文字（含「生成/画/做一张」等）
  → ① 分诊：image-query
  → ② 生图 Prompt 优化 → CogView 生图 API
  → ③ 成功话术模板 + 图片链接 → 返回
```

## 装备清单

| 项目 | 识图 | 生图 |
|------|------|------|
| 人设 Prompt | ✅ | ✅（优化描述 + 成功话术） |
| 工作流 | 2～3 步 | 2～3 步 |
| 数据库 | ❌ | ❌ |
| 工具 | 视觉大模型 | 生图 API |

## 成功标准

- 有图必走图片线，不误进文本检索
- 识图：先确认「看到图片」，再答问题
- 生图：返回可访问图片或明确失败提示

## 5 条测试题

| # | 输入 | 期望 |
|---|------|------|
| 1 | [上传图] 这是什么菜 | 识别并描述 |
| 2 | [上传图] 热量高吗 | 结合识图回答 |
| 3 | 生成一张宫保鸡丁的图片 | 返回图片或明确失败 |
| 4 | 画一份红烧肉 | 生图流程 |
| 5 | 生成宫保鸡丁图片（无上传） | 不生图误走识图 |

# ⑦ 文件业务线

## 是什么

用户 **上传文件**（txt/md/json/csv/xlsx 等），系统导入后可针对文件内容问答。

## 适用问题

- 带 file_path：「帮我分析这个菜谱」「总结上传的文档」
- Excel：走外部 **Ingest 服务** 导入 Postgres 等

## 工作流

```
上传文件 + 问题
  → ① 分诊：file-query（或有 file_path 强制进入）
  → ② 读文件 / Excel 调 ingest 服务
  → ③ 文本类：写入 Milvus 知识库
  → ④ 知识库检索 + 生成 → 返回
```

## 装备清单

| 项目 | 有没有 |
|------|--------|
| 人设 Prompt | ✅ 复用知识库生成 |
| 工作流 | ✅ 导入 + 检索 |
| 数据库 | ✅ 上传后 Milvus；Excel → Postgres ingest |
| 工具 | ✅ 文件解析、KnowledgeService、Ingest API |

## 成功标准

- 支持格式有明确提示；过大/file 不存在有友好报错
- 导入成功后，针对文件内容的提问能检索到相关内容

## 5 条测试题

| # | 输入 | 期望 |
|---|------|------|
| 1 | [上传 txt] 总结要点 | 导入并回答 |
| 2 | [上传 md] 有哪些菜名 | 基于文件内容 |
| 3 | 无 file_path「分析文件」 | 提示提供路径 |
| 4 | [上传 xlsx] 导入 | 触发 ingest（若已配置） |
| 5 | 不支持格式 | 明确不支持提示 |

# 附录 C：汇报用一句话（每条线）

1. **闲聊**：不查库，只陪聊。  
2. **追问**：问不清就先问用户。  
3. **文化库**：查文章库，讲历史不讲步骤。  
4. **做菜图谱**：查关系图，讲怎么做。  
5. **数据统计**：查 MySQL，报数字。  
6. **图片**：识图或生图。  
7. **文件**：先吃文件，再问答。
